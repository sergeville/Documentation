## Key Decisions

- Minutes remain markdown files in the repo as the source of truth.
- Archon should read and render minutes, not replace them as canonical storage.
- Minutes should support operator decision-making, not just historical logging.
- Reconciliation should surface related minutes evidence directly in context when possible.
- Future memory use should come from structured extraction out of minutes, not ad hoc manual interpretation every time.
- The inheritance program is only complete when repo truth and connected Archon truth agree.
- Idle cognition remains bounded background sensemaking and suggestion-only by default.
- Live registry schema is part of inheritance truth; runtime services without matching tables are not an acceptable completion state.
- The operator-facing inheritance UI must surface degraded fallback truth rather than fail when payloads are partial.
- MCP client visibility must distinguish between:
  - MCP server status
  - static client configuration instructions
  - backend-visible detected/configured client truth
- Archon should not imply host-local Codex or other assistants are detected unless the backend runtime can actually observe them or they self-register.
- The Codex setup instructions shown in the MCP page are canonical for configuration detection; once `[mcp_servers.archon]` exists in `~/.codex/config.toml`, Archon should mark Codex as configured.

## Risks / Gaps

- Minutes linkage is currently inferred by date, not by explicit story/task trace ids.
- Minutes are still read-only in Archon.
- The drift card currently shows summary-level minutes evidence, not section-level evidence extracted from the markdown body.
- Synapse still does not automatically promote unresolved minute actions into Archon state.
- Projects that have not been backfilled/reconciled still surface degraded inheritance fallback truth by design.
- The host-side migration runner currently depends on tooling (`asyncpg`) that was not available in this local environment, so the live inheritance migration was applied via direct `psql` instead of the Python runner.
- The current MCP client inventory is truthful only from the backend runtime boundary.
- Host-local Codex activity will still not appear in the MCP dashboard when Archon is running in a runtime that cannot inspect host processes or host config files.

## Recommended Next Step

1. Make inheritance migration application part of the standard local Archon bootstrap/check flow.
2. Start recording explicit story/task/sprint references in every new minutes file.
3. Implement structured extraction of minute actions into Archon memory.
4. Surface unresolved minute-derived actions in Archon operator views.
5. Implement host-truth publication or self-registration for MCP clients so Codex/Claude/Cursor can be shown as real operator-visible clients across runtime boundaries.
